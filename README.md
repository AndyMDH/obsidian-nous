# Cortex (Obsidian plugin)

The native Obsidian version of [Cortex](https://github.com/AndyMDH/cortex): turns
dictated or pasted notes into a linked, tagged knowledge graph, right inside
Obsidian. No CLI, no bash scripts, no `launchd` - the plugin calls the
Anthropic API directly and can enrich a note within seconds of you creating
it, instead of waiting for a once-a-day batch run.

Same folder layout and note schema as the original bash-based Cortex
(`00-Inbox` → `10-Meetings` → `20-Wikis`, `30-Tags` as a tag registry), so an
existing Cortex vault can adopt this plugin without migrating any data.

## Requirements

- Obsidian
- An [Anthropic API key](https://console.anthropic.com/settings/keys) - this
  is separate from a Claude subscription or Claude Code login, and is billed
  per token on your Anthropic Console account

## Install

Not on the Community Plugins list. Two options:

**BRAT (recommended)** - install the [BRAT](https://github.com/TfTHacker/obsidian42-brat)
community plugin, then "Add beta plugin" → `AndyMDH/obsidian-cortex`. BRAT
handles updates for you.

**Manual** - download `main.js`, `manifest.json`, and `styles.css` (if
present) from the [latest release](https://github.com/AndyMDH/obsidian-cortex/releases),
and copy them into `<vault>/.obsidian/plugins/cortex/`. Then enable "Cortex"
in Settings → Community plugins.

After enabling, open Settings → Cortex and paste in your API key.

## How it works

```
dictate/paste -> 00-Inbox/ -> enrich (API call) -> 10-Meetings/ -> wiki-builder (API call) -> 20-Wikis/
```

- **Enrichment**: watches `00-Inbox/` and, within a couple seconds of a new
  note appearing, sends it to Claude with your current tag registry and a
  compact index of recent notes (for duplicate detection and related-note
  linking). Claude returns structured data - the plugin does the actual file
  read/write itself. Your raw text is never regenerated by the model; it's
  copied verbatim under `## Transcript` by the plugin.
- **Wikis**: once a tag has enough non-fragment notes behind it (default: 4),
  a second call synthesizes a narrative wiki page in `20-Wikis/`, and the
  plugin patches the `## Related` section of every source note to link back
  to it.
- **Catch-up**: also runs a full inbox scan every time Obsidian starts, so
  anything captured while it was closed still gets processed.
- Manual trigger any time: ribbon icon, or command palette → "Cortex: Process
  inbox now" / "Cortex: Build/update wikis now".
- A `.cortex/pipeline.log` file in your vault logs every enrichment,
  duplicate, new tag, and wiki event, for the same reason the bash version
  kept one - so you can see what happened without re-reading every note.

## Settings

API key, model, wiki threshold, duplicate-check lookback (how many recent
notes to compare new captures against), auto-process on capture (on/off),
and the four folder paths - all configurable in Settings → Cortex.

## Limitations, honestly

- **Obsidian has to be running** to catch a new capture. There's no
  background daemon like the bash version's `launchd` job - the startup
  catch-up scan is the safety net, not a guarantee of same-day processing if
  Obsidian stays closed.
- **The API key lives in plaintext** in this vault's
  `.obsidian/plugins/cortex/data.json`. Keep this vault out of any repo or
  sync you don't fully control.
- **Single API call, not an agentic loop.** The original skill-based version
  let Claude Code re-read files and self-correct across multiple tool calls;
  this plugin asks for one structured response per note. Good enough for
  straightforward captures, less room to recover on genuinely ambiguous ones.
- **You need your own Anthropic API key and it's billed separately** from a
  Claude Pro/Max subscription or an existing Claude Code login.

## Development

```bash
npm install
npm run dev      # esbuild watch mode
npm run build    # typecheck + production bundle -> main.js
npm test         # unit tests for the enrichment/wiki logic (no live API calls)
```

The core logic (`src/`) has no dependency on the Obsidian runtime and is
unit-tested directly with Node's built-in test runner. `main.ts` wires that
logic to the actual `obsidian` API (vault I/O, settings UI, commands) and can
only be exercised inside a real Obsidian instance.
